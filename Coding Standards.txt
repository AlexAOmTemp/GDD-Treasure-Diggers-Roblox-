Coding Standards / Technical Guidelines – Treasure Diggers (Updated)
1. Общие принципы

Серверная авторитетность:

Все расчёты дохода, бонусов, прогресса и проверка условий выполняются только на сервере.

Клиент может отправлять только идентификаторы действий (ChestID, EggID, Rebirth trigger), не суммы и не валюту.

Чистый код:

Не использовать магические числа в скриптах.

Все числовые значения (цены, шансы, таймеры) берутся из соответствующих ModuleScripts/Config (ChestConfig, PetConfig, RebirthConfig, QuestConfig).

Понятные имена переменных и функций, однозначные префиксы (AddCoins, HatchEgg, FusePet).

Логирование и отладка:

Ошибки асинхронных операций должны логироваться с достаточной информацией (playerID, action, error).

Не использовать print для логики игры в релизе — только warn или Logger-модуль.

2. Data / Profile Management

Atomic Operations / isProcessing:

Каждая операция, изменяющая профиль игрока, должна использовать atomic lock (profile.isProcessing).

Scoped locks: флаг снимается в finally или через callback после завершения всех yield-операций.

Таймаут: если операция длится более 10 секунд, флаг принудительно сбрасывается с логированием. Для этого использовать task.delay с функцией сброса.

При загрузке профиля (PlayerAdded) isProcessing всегда устанавливается в false.

Обработка выхода игрока во время атомарной операции:
Если игрок покидает игру во время выполнения атомарной операции (isProcessing = true), флаг снимается сервером после таймаута 10 сек. При следующем входе профиль загружается в последнем сохранённом состоянии (до начала операции), так как изменения не были зафиксированы. Это безопасно, но может привести к потере монет, если операция уже частично выполнена (но мы гарантируем атомарность). Рекомендуется логировать такие случаи для последующего анализа.

DataMigrationService:

Во время миграции (profile.isMigrating = true) профиль должен быть read-only для всех менеджеров.

Все менеджеры должны проверять isMigrating перед изменением данных.

Старая версия данных не должна перезаписывать новые поля.

Перед миграцией создаётся резервная копия данных; при ошибке данные восстанавливаются.

Guard Clauses:

В начале каждой функции проверки состояния профиля:

if not profile or profile.isProcessing or profile.isMigrating then return end

Session Lock + Retry:

Все операции с DataStore используют session lock и pcall для защиты от временных сбоев. Реализовать механизм повторных попыток (до 3 раз) с экспоненциальной задержкой.

Bulk операции (например, Bulk Open) должны быть атомарными с полным откатом при сбое.

3. RemoteEvents / Networking

Минимизация payload:

Отправлять только необходимые данные (IDs, минимальный набор полей для UI).

PetStatUpdate для одного питомца, PetsInventoryUpdate — полный синхрон.

Batch / пакетная отправка:

Использовать буферизацию для частых событий (0.1–0.2 сек) через task.defer или Heartbeat цикл.

Исключает сетевой шторм и повышает производительность.

Для ChestHPUpdate использовать батчинг, чтобы не отправлять каждый клик.

Security:

Проверка ID действий и лимитов полностью на сервере.

DistanceCheck, ClickRateLimit и Atomic Claim для сундуков.

Внедрить RateLimit на каждое действие от клиента (например, не более 10 запросов в секунду). RateLimitService применяется ко всем входящим событиям и не зависит от клиентских ограничений.

Защита от повторной отправки одного и того же ChestID (проверять, не был ли уже обработан).

Для Rebirth и RareAreaAccess ввести дополнительные кулдауны (10 сек и 1 сек соответственно). Кулдауны хранятся в памяти сервера и не сохраняются в профиль. Это допустимо, так как смена сервера — редкое событие, а кулдауны небольшие.

4. Managers / Modules

CoinsManager

AFK / Friends bonus проверяются сервером. Максимальный бонус 100% (конфиг). AFK Income начисляется только в специальной AFK-зоне, при этом игрок должен быть неактивен (отсутствие кликов/перемещений) в течение 10 секунд.

Не изменять профиль напрямую, использовать атомарные функции с проверкой isProcessing.

ChestManager / MegaChestManager

Atomic Claim + ClickRateLimit + DistanceCheck обязательны. Учитывать пинг (буфер истории позиций за 0.5 сек или допуск distance + speed*ping).

HP сундуков через Attributes и GetAttributeChangedSignal. Обновления отправляются не чаще 0.2 сек. Для Mega Chest урон обрабатывается атомарно (очередь/блокировка) с применением Damage Cap (50% HP за интервал).

PetsManager / Fusion / Golden Pets

Fusion Threshold проверяется сервером.

Golden Pets получают дополнительный множитель.

XP и Level Up выполняются на сервере.

При Fusion новый питомец наследует максимальный уровень и максимальную XP (внутри этого уровня) из исходных питомцев.

Автоудаление: при добавлении питомца проверять настройки игрока и, если включено, удалять лишних по приоритету: Common → Rare → низкий уровень → старый. Если после удаления всех возможных питомцев места всё ещё недостаточно, операция отклоняется с ошибкой "Inventory full".

QuestManager / RareAreaManager / RebirthManager

Все проверки лимитов, прогресса и таймеров на сервере.

HUD обновляется через RemoteEvents.

Для RareAreaManager добавить кулдаун на вход/выход (1 сек).

Для RebirthManager добавить кулдаун на запрос Rebirth (10 сек).

5. Object Pool / Optimization

Использовать Object Pool для сундуков, VFX, Floating Numbers.

Ограничить количество одновременно активных Floating Numbers до 10.

OnRelease сбрасывает все атрибуты (HP, бонусы, статус).

StreamingEnabled и LOD для производительности.

6. Technical / Security Notes

Anti-cheat:

Проверка позиции игрока при AFK Income и Rare Area (на каждом входе в зону).

Atomic Claim, DistanceCheck, ClickRateLimit с абсолютным hard cap, проверка equippedPets при Fusion/Rebirth.

Защита от спама RemoteEvents через RateLimitService.

Performance:

Bulk Open / Pity System, batch обновления UI.

Ограничение Draw Calls, оптимизация RemoteEvents, Mobile Safe Zone для HUD/UI.

7. Implementation Notes
7.1 AtomicAction + isProcessing

Флаг снимается только после полного завершения всех yield-операций.

Для асинхронных операций использовать callback или Promise-подобный механизм.

Добавить таймаут 10 сек: task.delay(10, function() if profile.isProcessing then profile.isProcessing = false; warn("Timeout on atomic action") end end)

При входе игрока сбрасывать isProcessing и isMigrating.

7.2 Batch / пакетная отправка

Для частых событий (PetStatUpdate, QuestProgressUpdate) использовать очередь с task.defer или Heartbeat.

Отправка всех изменений за один пакет каждые 0.1–0.2 сек.

Для ChestHPUpdate также использовать батчинг.

7.3 DataMigrationService

Устанавливать profile.isMigrating = true на время миграции.

Все менеджеры должны проверять флаг и приостанавливать изменения профиля.

Старые менеджеры не должны перезаписывать новые поля.

Пример: при миграции с v3 на v4 добавить поле quests, заполнить значениями по умолчанию.

Перед миграцией создавать резервную копию данных. При ошибке восстанавливать из резерва.

7.4 Guard Clauses

В начале каждой функции менеджера:

if not profile or profile.isMigrating or profile.isProcessing then return end

7.5 Таймауты и логирование

Если isProcessing держится дольше 10 секунд → сброс + warn("Timeout on atomic action").

Все ошибки асинхронных операций логировать для отладки.

7.6 Кулдауны

Кулдауны на действия (Rebirth, вход в Rare Area) хранятся в памяти сервера и не сохраняются в профиль. Это достаточная защита от спама в рамках одной сессии. При переходе на другой сервер игрок сможет выполнить действие сразу, что приемлемо для данных механик.

8. Config / No Magic Numbers

Все числа (цены, шансы, таймеры, мультипликаторы) берутся из ModuleScripts.

Config может быть JSON, ModuleScript или таблица Lua.

Добавить конфиг для RateLimit: maxRequestsPerMinute = 60 и т.д.

Добавить конфиг для кулдаунов: RebirthCooldown = 10, RareAreaTriggerCooldown = 1.

Таблица констант:
Все жёсткие ограничения (AbsoluteMaxCPS = 20, MaxFloatingNumbers = 10, AFKCheckInterval = 5 сек и т.д.) должны быть вынесены в ModuleScripts/Config/GlobalConstants. Геймдизайнер может менять их без изменения логики менеджеров.
